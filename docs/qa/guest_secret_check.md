# 動作確認手順: ゲスト出欠回答の guest_secret クレームモデル

対象: `supabase/migrations/20260713000000_add_guest_secret.sql` と、それに伴う
`app/api/share/plan/[token]/identify`, `app/api/share/plan/[token]/attendance`,
`app/api/manual-plans/[id]/members/[memberId]/reset-guest-secret`,
`components/share/AttendanceForm.tsx`, `components/manual-plans/MemberGuestSecretReset.tsx`
の変更確認。

## 前提

1. マイグレーションを適用する:
   ```bash
   supabase db push
   ```
   (このタスクでは未実行。確認作業を始める前に実行すること)
2. 開発サーバを起動する:
   ```bash
   rm -rf .next && npm run dev
   ```
3. 通常のアカウントでログインしておく(幹事役)。

## 手順

### 1. プラン作成 → メンバー追加 → 共有URL取得

1. `/manual-plans/new` でプランを作成し、メンバーを2人追加する(例: 「Aさん」「Bさん」)。
2. 保存後、プラン詳細ページ(`/manual-plans/[id]`)を開き、「URLで共有」で共有URL(`/share/plan/[token]`)をコピーする。

### 2. メンバーAが出欠登録できることを確認

1. シークレット(プライベート)ウィンドウを開き、共有URLにアクセスする。
2. 「あなたのお名前を選んでください」から **Aさん** を選択する。
3. 「参加する」を選び、正常に更新される(ボタンがハイライトされ、エラーが出ない)ことを確認する。
4. ブラウザの開発者ツール → Application → Session Storage で以下のキーが保存されていることを確認する:
   - `kanjii_attendance_member_[token]` = Aさんのmember_id
   - `kanjii_guest_secret_[token]_[AさんのID]` = UUID形式の値

### 3. 同じメンバーAを別セッションから選ぶとエラーになることを確認

1. **別の**シークレットウィンドウ(1と別プロセス/別セッションになるよう新規に開く)で同じ共有URLを開く。
2. 「あなたのお名前を選んでください」から再度 **Aさん** を選択する。
3. 「このメンバーは既に他の端末で登録済みです。幹事の方に問い合わせてください。」という趣旨のエラーメッセージが表示され、名前選択画面に戻ることを確認する。
4. Networkタブで `POST /api/share/plan/[token]/identify` が **409** を返していることを確認する。

### 4. メンバーBは独立して出欠登録できることを確認

1. 手順3と同じ(または新規の)シークレットウィンドウで、今度は **Bさん** を選択する。
2. 「不参加」等を選び、正常に更新できることを確認する(Aさんの回答には影響しない)。
3. プラン詳細ページ(幹事側)をリロードし、Aさん=参加、Bさん=不参加と正しく別々に反映されていることを確認する。

### 5. なりすまし更新ができないことを確認(重要: セキュリティ修正の本題)

1. ブラウザの開発者ツールでNetworkタブを開いた状態で、手順4のBさんのタブから `PATCH /api/share/plan/[token]/attendance` のリクエストをコピーする(または手動で `fetch` を叩く)。
2. `member_id` をAさんのIDに書き換え、`guest_secret` はBさんの値のまま(またはでたらめな値)にしてリクエストを送る。
3. **403** が返り、Aさんの出欠が変更されていないことを確認する(修正前はここが200で通ってしまっていた)。

### 6. 幹事によるリセット導線を確認

1. 幹事アカウントでプラン詳細ページ(`/manual-plans/[id]`)を開く。
2. Aさんの行に「回答をリセット」ボタンが表示されていることを確認する(登録済みメンバーのみ表示される想定)。
3. クリックして「リセット済み」表示に変わることを確認する。
4. 別のシークレットウィンドウで共有URLを開き、Aさんを選び直せる(409エラーにならない)ことを確認する。
5. 新しい `guest_secret` が発行され、以降の出欠更新が正常に行えることを確認する。

## 期待結果まとめ

| 確認項目 | 期待結果 |
|---|---|
| 自分のメンバー行への出欠登録 | 成功する |
| 別セッションから同じメンバーを再度選択 | 409で拒否され、名前選択画面に戻る |
| 複数メンバーの独立性 | 互いの回答に影響しない |
| member_id改ざんによる他人の出欠書き換え | 403で拒否される |
| 幹事によるリセット | 対象メンバーが再度identify可能になる |

## 既知の制約(設計上の想定内)

- 同一人物が自分のセッションストレージを消してしまった場合も「他の端末」と同じ409になる。これは意図的な仕様(なりすまし防止のための「先着1名」ルール)であり、幹事によるリセットが唯一の回復手段。
